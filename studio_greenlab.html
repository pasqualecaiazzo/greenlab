Cos'è il progetto

GreenLab è un sito PHP con database MySQL per la consultazione di un catalogo di piante all'ingrosso. L'accesso è riservato agli operatori registrati (fioristi, garden center, rivenditori). C'è anche un'area amministratore separata per gestire il catalogo e gli utenti.

L'amministratore usa credenziali hardcoded nel codice (admin / admin1234), come richiesto dai requisiti ("credenziali predefinite"). La sessione admin è separata da quella utente.

Il database

Database: greenlab — 2 tabelle.

Tabella utenti

CREATE TABLE utenti (
  username VARCHAR(100) NOT NULL,
  cognome  VARCHAR(100) NOT NULL,
  email    VARCHAR(255) NOT NULL UNIQUE,
  password VARCHAR(255) NOT NULL,
  PRIMARY KEY (username)
);
  • Chiave primaria: username — come richiesto dai requisiti
  • email ha UNIQUE → non possono esistere due utenti con la stessa email
  • Solo i campi richiesti dai requisiti — niente telefono, ragione sociale, partita IVA

Tabella piante

CREATE TABLE piante (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  nome        VARCHAR(200)  NOT NULL,
  tipo        VARCHAR(100)  NOT NULL,
  descrizione TEXT          NOT NULL,
  prezzo      DECIMAL(10,2) NOT NULL
);
  • id generato automaticamente (AUTO_INCREMENT)
  • tipo è il campo usato per raggruppare le categorie — fondamentale per la ricerca a due passi
  • DECIMAL(10,2) = massimo 10 cifre totali, 2 decimali

La ricerca a due passi

Si chiama "a due passi" perché prima si sceglie la categoria, poi si scelgono le piante. Il dettaglio è il risultato finale, non un passo di ricerca.

Passo 1 — ricerca1.php: scelta categoria

$sql_categorie = "SELECT DISTINCT tipo FROM piante ORDER BY tipo";
$ris           = mysqli_query($conn, $sql_categorie);

SELECT DISTINCT restituisce ogni categoria una sola volta. Il menu si aggiorna automaticamente — non è hardcoded. Form con metodo GET: manda a ricerca2.php?tipo=Piante+da+interno.

Concetti chiave da saper spiegare

Le sessioni PHP

Mantengono dati tra una pagina e l'altra. Il server crea un ID univoco per ogni utente, salvato in un cookie nel browser. I dati stanno lato server.

  • session_start() — obbligatorio all'inizio di ogni pagina che usa le sessioni, prima di qualsiasi output HTML
  • $_SESSION["chiave"] = valore — salva un dato
  • isset($_SESSION["chiave"]) — controlla se esiste
  • session_destroy() — elimina tutta la sessione (logout)

Flusso delle query MySQL in PHP

$sql = "SELECT * FROM piante WHERE tipo = \"Succulente e Cactus\"";
$ris = mysqli_query($conn, $sql);       // esegue
mysqli_num_rows($ris);                  // quante righe
while ($riga = mysqli_fetch_assoc($ris)) {
    echo $riga["nome"];                 // legge riga per riga
}

GET vs POST

  • GET: dati nell'URL (ricerca2.php?tipo=Lavanda). Usato per ricerca1→ricerca2 perché la categoria è nell'URL e si può navigare col tasto indietro.
  • POST: dati nel corpo della richiesta, invisibili nell'URL. Usato per login, registrazione, e tutte le operazioni che modificano il database (INSERT, UPDATE, DELETE).

$_SERVER["REQUEST_METHOD"] == "POST"

Molte pagine gestiscono sia la visualizzazione del form (GET) che l'elaborazione dei dati (POST). Questo controllo distingue i due casi: se la pagina è stata aperta normalmente mostra il form, se è stata inviata elabora i dati.

Il cast a intero (int)

$id = (int)$_POST["id"];

Converte l'id in intero prima di usarlo in una query. Garantisce che sia sempre un numero, eliminando qualsiasi testo o carattere speciale al posto di un id.

SELECT DISTINCT

SELECT DISTINCT tipo FROM piante ORDER BY tipo

Restituisce ogni valore di tipo una sola volta, anche se ci sono molte piante della stessa categoria. Il menu categorie si aggiorna automaticamente — non è hardcoded.

Perché exit dopo header()

header() da solo non ferma il codice PHP — imposta solo l'intestazione HTTP di redirect. Senza exit, tutto il codice successivo continua ad essere eseguito. exit ferma tutto immediatamente.

Perché isset() e non solo if()

isset() controlla sia che la variabile esista sia che non sia null. Se usassi if($_SESSION["username"]) e la variabile non esistesse, PHP genererebbe un warning. Con isset() il controllo è sicuro.

Domande all'orale

Clicca su ogni domanda per vedere la risposta.

Come funziona il login?

L'utente inserisce username e password nel form. Il PHP fa una query che cerca nella tabella utenti un record con quell'username E quella password. Se trova esattamente 1 riga, salva lo username nella sessione e reindirizza alla home. Altrimenti mostra un messaggio di errore.

Come impedisci a un utente non loggato di accedere alle pagine riservate?

All'inizio di ogni pagina riservata chiamo controllaSessione(). Quella funzione controlla se esiste $_SESSION["username"]. Se non esiste, fa un redirect a login.php e si ferma con exit. Senza exit, il resto del codice continuerebbe ad essere eseguito.

Come funziona la ricerca a due passi?

Prima l'utente sceglie una categoria da un menu generato dinamicamente con SELECT DISTINCT tipo. Poi vede la lista delle piante di quella categoria con le checkbox, ne sceglie una o più e gli id selezionati vengono salvati nella sessione. Nella pagina di dettaglio, per ogni id viene fatta una query e si mostra la scheda completa con nome, categoria, descrizione e prezzo.

Come fa l'admin ad aggiornare una pianta?

La pagina ha due fasi. Prima mostra la lista di tutte le piante con radiobox. Quando si preme "Modifica", fa una query per recuperare i dati attuali e mostra un form precompilato. Quando si preme "Salva" esegue un UPDATE nel database con i nuovi valori. Le due fasi si distinguono controllando quale pulsante è stato premuto con isset($_POST["scegli"]) e isset($_POST["salva"]).

Perché le password non sono criptate?

Non è stato trattato a lezione. So che PHP ha funzioni come password_hash() e password_verify() per gestire l'hashing in modo sicuro, ma non rientrava nel programma del corso.

Cosa succede se un utente cerca di accedere a una pagina admin?

controllaSessioneAdmin() controlla se esiste $_SESSION["admin"]. Se non esiste, reindirizza a login.php e blocca l'esecuzione con exit. Senza sessione admin attiva, nessuna pagina del pannello è accessibile.

Come gestisci il caso in cui nessuna pianta venga trovata?

In ricerca2.php controllo mysqli_num_rows($ris). Se è 0 mostro il messaggio "Nessuna pianta trovata per questa categoria" e un link per tornare al passo 1. La stessa logica si applica nelle pagine admin quando non ci sono piante o utenti da gestire.

Cosa fa session_destroy()?

Elimina completamente la sessione sul server — tutti i dati salvati in $_SESSION vengono cancellati. Dopo questa chiamata l'utente non è più autenticato. Il cookie nel browser rimane ma è inutile perché i dati lato server non esistono più.

Perché usi GET per la ricerca categoria e POST per il login?

Con GET la categoria finisce nell'URL, quindi l'utente può tornare indietro col browser e la pagina è navigabile normalmente. Per il login uso POST perché le credenziali non devono essere visibili nell'URL e non devono essere reinviate se si aggiorna la pagina.

Come funziona il controllo della chiave primaria username?

In fase di registrazione faccio una SELECT per verificare che l'username non sia già presente. Se mysqli_num_rows() restituisce un valore maggiore di 0, l'username è già in uso e mostro un errore. La PRIMARY KEY nel database garantisce anche a livello di database che non esistano duplicati.
